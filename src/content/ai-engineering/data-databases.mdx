## Why Data Architecture Matters

AI systems fail when their data layer is vague. A model can only be as reliable as the data contracts, metadata, indexing strategy, and observability around it.

## Storage Layers

### Transactional Store

Use PostgreSQL or another OLTP database for users, permissions, plans, documents, jobs, and durable application state.

### Analytical Store

Use warehouses, lakehouses, or columnar engines for aggregate analysis, experiments, and model monitoring.

### Vector Store

Use pgvector, Qdrant, Weaviate, or similar systems for embedding search. The vector store should not be treated as the source of truth. It is an index over content.

### Cache and Queue

Redis is useful for caching, rate limits, sessions, background jobs, and short-lived state.

## Retrieval Data Model

For RAG, store:

- Document ID
- Chunk ID
- Source URL or file path
- Content hash
- Chunk text
- Metadata
- Embedding model version
- Created and updated timestamps
- Access-control fields

## Failure Modes

- Re-indexing without tracking embedding model version.
- Losing the relationship between chunks and source documents.
- Mixing public and private documents without access filters.
- Caching answers that depend on user-specific permissions.
- Treating vector similarity as truth instead of candidate retrieval.

## Production Principle

Keep raw source, parsed text, chunks, embeddings, and generated answers as separate artifacts. That separation makes debugging possible.
